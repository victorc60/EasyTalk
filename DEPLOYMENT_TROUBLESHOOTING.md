# Решение проблем при деплое

## Почему проблема с БД могла возникнуть внезапно

Возможные причины, если раньше всё работало:

1. **Railway отдаёт другие переменные**  
   Сервис MySQL на Railway даёт `MYSQL_URL`, `MYSQLHOST`, `MYSQLPORT`, `MYSQLUSER`, `MYSQLPASSWORD`, `MYSQLDATABASE`, а не `DATABASE_URL`. Если раньше вы вручную задавали `DATABASE_URL`, а потом пересоздали сервис или перепривязали переменные, приложение могло перестать видеть строку подключения.  
   **Решение:** приложение теперь поддерживает и `DATABASE_URL`, и `MYSQL_URL`, и сборку URL из `MYSQLHOST`/`MYSQLUSER`/`MYSQLPASSWORD`/`MYSQLDATABASE`. Убедитесь, что у сервиса приложения есть ссылка на переменные MySQL (Reference → выбрать MySQL-сервис) или скопирована хотя бы одна строка: `DATABASE_URL` или `MYSQL_URL`.

2. **Перезапуск или обновление контейнера БД**  
   После деплоя или обновления MySQL-сервиса контейнер поднимается 1–3 минуты. Приложение раньше падало при первой же ошибке подключения.  
   **Решение:** добавлены повторные попытки при старте (`DB_RETRY_ATTEMPTS`, `DB_RETRY_DELAY_MS`).

3. **Жёсткая проверка переменных**  
   Раньше в `config.js` требовались сразу `DATABASE_URL`, `DB_HOST`, `DB_NAME`, `DB_USER`, `MYSQLPASSWORD`. На Railway при связке с MySQL часто появляется только `MYSQL_URL` (или переменные `MYSQL*`), из-за чего приложение падало при загрузке конфига.  
   **Решение:** конфиг считает БД настроенной, если задан любой из вариантов: `DATABASE_URL`, `MYSQL_URL` или полный набор `MYSQL*` / `DB_*`.

4. **Таймаут или «зависшее» подключение**  
   При долгом старте БД или нестабильной сети первая попытка подключения могла висеть и падать по таймауту.  
   **Решение:** в Sequelize заданы `connectTimeout` и настройки пула (`pool.acquire`, `pool.idle`).

**Диагностика:** локально (с тем же `.env` или переменными) можно запустить:
```bash
node scripts/check-db.js
```
Скрипт покажет, какие переменные БД заданы (без паролей) и удаётся ли подключиться.

## Возможные проблемы и решения

### 0. Railway: "We are unable to connect to the database via SSH" / "Database container is starting up"

**Симптомы:** В панели Railway при открытии сервиса базы данных показывается, что подключение по SSH недоступно, контейнер «запускается или переходит в другой режим».

**Что это значит:** Это сообщение относится к **интерфейсу Railway** (просмотр БД через их UI), а не обязательно к вашему приложению. Приложение подключается к MySQL по `DATABASE_URL` по внутренней сети; панель подключается по SSH отдельно и может быть недоступна, пока контейнер БД ещё стартует.

**Что делать:**
1. Подождать 2–5 минут и снова нажать подключение в панели БД.
2. Проверить, что бот реально работает (написать боту в Telegram).
3. В Railway: сервис приложения → **Deployments** → **View Logs** — искать строку `✅ База данных подключена`. Если она есть, приложение к БД подключено, ошибка только в UI.
4. Убедиться, что у сервиса приложения задана переменная подключения к MySQL: либо `DATABASE_URL`, либо `MYSQL_URL` (Railway для MySQL даёт именно `MYSQL_URL` и набор `MYSQLHOST`, `MYSQLUSER`, `MYSQLPASSWORD`, `MYSQLDATABASE`). Свяжите сервис приложения с MySQL через **Variables → Add Reference** и выберите нужные переменные, либо скопируйте `MYSQL_URL` в переменные сервиса приложения.

В коде добавлены повторные попытки подключения к БД при старте (`DB_RETRY_ATTEMPTS`, `DB_RETRY_DELAY_MS`), чтобы приложение не падало, пока MySQL на Railway ещё поднимается.

### 0.1. Telegram: 409 Conflict / "terminated by other getUpdates request"

**Симптомы:** В логах: `ETELEGRAM: 409 Conflict: terminated by other getUpdates request; make sure that only one bot instance is running`.

**Причина:** Одновременно работают **два или больше экземпляров** одного и того же бота в режиме **polling** (getUpdates). Telegram разрешает только один активный long polling на один токен.

**Что делать:**
1. **Запущен только один экземпляр:** не запускайте бота и локально, и на Railway одновременно в режиме polling. Либо останавливайте локальный процесс, когда деплой на Railway уже работает (или наоборот).
2. **На продакшене (Railway) лучше использовать webhook:** в переменных сервиса задайте `TELEGRAM_WEBHOOK_URL` или `WEBHOOK_DOMAIN` (ваш публичный URL приложения, например `https://your-app.railway.app`). Тогда на Railway бот будет получать обновления по webhook, а не по polling, и конфликта не будет. Локально можно оставить polling для разработки (но не запускать локальный и Railway одновременно с одним токеном и оба на polling).
3. В коде polling теперь **включается только после** успешного подключения к БД и настройки бота, чтобы не создавать лишних запросов getUpdates до готовности приложения.

### 1. Проблема: Ошибка при запуске контейнера

**Симптомы:** Сборка проходит успешно, но контейнер не запускается или падает сразу после старта.

**Решения:**
- Проверьте переменные окружения (`.env` файл не копируется в образ)
- Убедитесь, что все необходимые переменные установлены в среде деплоя:
  - Обязательно: `TELEGRAM_BOT_TOKEN`, `OPENAI_API_KEY`, `ADMIN_ID`
  - Для БД — один из вариантов:
    - `DATABASE_URL` (полная строка), или
    - `MYSQL_URL` (Railway MySQL), или
    - `MYSQLHOST`, `MYSQLUSER`, `MYSQLPASSWORD`, `MYSQLDATABASE` (Railway), или
    - `DB_HOST`, `DB_NAME`, `DB_USER`, `MYSQLPASSWORD`
- Проверьте логи контейнера после запуска:
  ```bash
  docker logs <container_id>
  ```

### 2. Проблема: Отсутствие package-lock.json

**Симптомы:** Ошибка `npm ci` или предупреждения о несоответствии зависимостей.

**Решение:** Обновленный Dockerfile теперь автоматически использует `npm install`, если `package-lock.json` отсутствует.

### 3. Проблема: Недостаточно места на диске

**Симптомы:** Ошибки при копировании файлов или экспорте образа.

**Решение:** 
- Очистите неиспользуемые образы: `docker system prune -a`
- Проверьте доступное место на диске

### 4. Проблема: Уязвимости в зависимостях

**Симптомы:** Предупреждения `npm audit` о уязвимостях.

**Решение:** 
- Это предупреждения, а не ошибки
- Для продакшена можно игнорировать (флаги `--no-audit --no-fund` уже добавлены)
- Для обновления зависимостей запустите локально: `npm audit fix`

### 5. Проблема: Отсутствие необходимых файлов

**Симптомы:** Ошибки при запуске типа "Cannot find module" или "File not found".

**Решение:**
- Проверьте `.dockerignore` - убедитесь, что важные файлы не исключены
- Убедитесь, что все директории копируются (кроме `node_modules`, `.env`, `.git`)
